# Design: iam-agent-proxy

## Problem statement

AWS IAM Identity Center (IAC) roles cannot be modified. Roles under `/aws-reserved/` return `UnmodifiableEntity` on `UpdateAssumeRolePolicy`. The trust policy allows only `sts:AssumeRoleWithSAML` from the SAML provider — self-assumption is not possible. This means an agent cannot directly assume an IAC role, and session policies (which require an intermediate `AssumeRole` call that the role's trust policy would have to permit) are also unreachable.

The proxy is the workaround: it holds real AWS credentials (via boto3 in the native path, or elhaz in the Docker path) and re-signs outbound requests. The agent never holds IAC credentials; it holds credentials that authenticate to the proxy.

## Credential carrier pattern

The agent holds a fake-but-syntactically-valid AWS keypair generated by the proxy. There is no IAM user, no IAM identity, nothing in AWS that corresponds to these keys. The keys exist solely so the AWS SDK will build and sign a request — the SDK refuses to proceed without `AWS_ACCESS_KEY_ID` and `AWS_SECRET_ACCESS_KEY` present and syntactically valid.

The proxy generates the keypair at startup (or per agent session), holds the secret in memory, and validates inbound SigV4 signatures locally. If the keys appear in a prompt, a log, or an exfiltrated file, they are useless: AWS will reject them immediately because no IAM identity backs them. The proxy is the only entity that can do anything with them.

This separates two concerns IAM conflates: *who is making this request* (the proxy-issued keypair) and *what AWS actions are permitted* (the upstream credential session). Credential leakage from the agent only compromises its identity to the proxy.

## Upstream credential sources

### Native path (boto3)

In the native (non-Docker) path, `core/upstream_creds.py` fetches real credentials from a named AWS profile:

```python
class BotoCredentialSource:
    def __init__(self, profile_name: str = "iam-agent-proxy") -> None:
        self._session = boto3.Session(profile_name=profile_name)

    def get(self) -> Credentials:
        creds = self._session.get_credentials().get_frozen_credentials()
        return Credentials(creds.access_key, creds.secret_key, creds.token)
```

A single `boto3.Session` is held for the lifetime of the proxy. This is important: botocore's `RefreshableCredentials` machinery (used by SSO, instance profiles, assumed roles) lives on the session's credential resolver object. Re-creating the session on each request would throw away that state and force full provider-chain re-discovery on every AWS call. With a persistent session, botocore can refresh expiring credentials transparently.

The profile defaults to `iam-agent-proxy` so the proxy never silently consumes `[default]` credentials. Override with the `AWS_PROXY_PROFILE` env var. Any standard credential mechanism works: SSO, `credential_process`, static keys, instance profiles.

### Docker path (elhaz)

In the Docker-compose path, `core/elhaz.py` fetches credentials from the [elhaz](https://github.com/61418/elhaz) daemon via a Unix socket. elhaz manages short-lived IAC credentials for IAM Identity Center roles. This source is used only in the Docker path and is not imported by the native proxy code.

## Credential vending via Unix socket

The proxy generates a keypair and serves it to the agent over a Unix socket on a shared Docker volume (or local path). The agent consumes it via the AWS `credential_process` mechanism — no custom client code required, the SDK calls a small helper binary that connects to the socket and prints the expected JSON.

**Proxy side** — a thread running alongside the proxy.py plugin (`core/credentials.py`):

```python
import socket, json, os, secrets, threading

def serve_creds(sock_path, get_creds):
    if os.path.exists(sock_path):
        os.unlink(sock_path)
    with socket.socket(socket.AF_UNIX, socket.SOCK_STREAM) as s:
        s.bind(sock_path)
        s.listen()
        while True:
            conn, _ = s.accept()
            with conn:
                conn.sendall(json.dumps(get_creds()).encode())
```

`get_creds()` returns the current keypair plus an `Expiration` timestamp. The proxy regenerates the keypair on rotation and updates what `get_creds()` returns; clients fetching after rotation get new material automatically.

**Agent side** — `~/.aws/config` in the agent container:

```ini
[profile proxy]
credential_process = /usr/local/bin/proxy-creds
```

`proxy-creds` is a small script that connects to the socket and prints the credential JSON the SDK expects:

```json
{
  "Version": 1,
  "AccessKeyId": "AKIAPROXY0000000001",
  "SecretAccessKey": "...",
  "Expiration": "2026-04-29T01:00:00Z"
}
```

The SDK caches credentials until `Expiration` and calls `proxy-creds` again when they're near expiry. Rotation requires no restart on either side.

## CA certificate management (native path)

In the native path, the proxy generates a self-signed RSA-4096 CA certificate on first run and stores it under `~/.iam-agent-proxy/`:

```
~/.iam-agent-proxy/
  ca.pem   # CA certificate (10-year validity, IsCA=true)
  ca.key   # CA private key (chmod 600)
```

On startup the proxy writes `ca_bundle = ~/.iam-agent-proxy/ca.pem` into `[default]` in `~/.aws/config`. This tells the AWS CLI and SDKs to trust the proxy's CA without requiring `AWS_CA_BUNDLE` to be set in every shell. The entry is removed on clean exit (SIGINT/SIGTERM).

In the Docker path, the CA cert is generated by the proxy container and shared with the agent container via a named volume.

## Local SigV4 validation

The proxy validates inbound requests locally without calling AWS. SigV4 is HMAC-SHA256 over a canonical request string derived from the method, host, path, query, headers, and body hash. The proxy holds the current secret key (the same one it vended via `creds.sock`) and recomputes the expected signature with no network round-trip.

The validation flow:
1. Proxy receives a request signed with the proxy-issued keypair.
2. Proxy verifies the HMAC-SHA256 signature locally using the held secret.
3. On success, proxy strips all SigV4 headers (`Authorization`, `X-Amz-Date`, `X-Amz-Security-Token`, `X-Amz-Content-Sha256`).
4. Proxy fetches fresh credentials from the upstream source (boto3 or elhaz).
5. Proxy recomputes SigV4 using those credentials and adds the new headers.
6. Signed request is forwarded to the AWS endpoint.

During a rotation, the proxy briefly holds both the old and new secrets so in-flight requests signed with the old key still validate.

## proxy.py as the HTTPS MITM layer

The native path uses [proxy.py](https://github.com/abhinavsingh/proxy.py) for TLS interception. The plugin class `core/addon.py:ResignPlugin` extends `proxy.http.proxy.plugin.HttpProxyBasePlugin` and implements `handle_client_request(request: HttpParser)`.

`proxy.py` has zero base dependencies, so the entire native stack installs as:

```bash
pip install proxy.py botocore boto3 cryptography pydantic
```

No separate venvs, no subprocess wrappers, no dependency conflicts.

In the Docker path, the proxy container also uses proxy.py. The `core/elhaz.py` credential source is installed into a separate venv (`/opt/elhaz-venv`) to isolate elhaz's typing-extensions requirements from proxy.py. This isolation is only needed in Docker; native installs do not use elhaz at all.

## Docker isolation as enforcement boundary

The proxy runs on the host (or in a privileged sidecar). IAC credentials and the elhaz socket are only accessible in that context.

Untrusted agent processes run in a container with:
- No host network access (`--network none` or a dedicated bridge with no host route)
- No elhaz socket mount
- No IAM instance profile or credential file accessible from inside the container

The agent container gets only `creds.sock` (to fetch proxy keypair material) and the proxy.py port (to send AWS traffic through). It cannot reach elhaz, cannot reach the IAC credentials, and cannot reach AWS directly.

Credential leakage from within the container is harmless: the proxy-issued keypair is meaningless to AWS, and `creds.sock` only vends the same keypair the agent already has.

This makes isolation a property of the environment, not a property of the agent. The agent does not need to be trustworthy about credential handling. It cannot misuse credentials it does not hold.

## Recording and enforcement modes

The proxy operates in two modes.

### Recording mode

All validated requests are forwarded to AWS. The proxy resolves each request to IAM action strings (using the same logic as enforcement mode) and logs them. Running the agent against a representative workload produces a behavioral profile — what the agent actually called, not what someone guessed it would need.

### Enforcement mode

Set `PROXY_MODE=enforce` and `ALLOWLIST_PATH=/path/to/policy.json`. The proxy resolves each validated request to IAM action strings, checks them against the allowlist, and blocks anything not permitted:
- Match: forward and re-sign.
- No match: block and return a forged `AccessDenied` 403 (see [Error forgery](#error-forgery-for-blocked-requests)).

The allowlist is standard IAM policy JSON (`{"Version":"2012-10-17","Statement":[...]}`), usable directly with `aws iam put-role-policy` or as a session policy.

**Action resolution** — the proxy resolves HTTP requests to IAM action strings in-process with no external dependencies. `core/resolver.py` dispatches on the AWS wire protocol for each service:

- **`json` protocol** (DynamoDB, KMS, Lambda, etc.): reads the operation name from the `X-Amz-Target` header.
- **`query`/`ec2` protocol** (IAM, STS, EC2, SQS, etc.): reads the `Action` field from the URL-encoded POST body.
- **`rest-json`/`rest-xml` protocol** (S3, ECS, CloudWatch, etc.): matches the HTTP method and URI path against pre-compiled regex patterns derived from botocore service models. Disambiguation uses literal query string keys (e.g. `?acl`, `?tagging`), required query parameters (e.g. `partNumber` for `UploadPart`), and required headers (e.g. `x-amz-copy-source` for `CopyObject`).

Resolved operation names are looked up in `docs/map.json` (vendored from `iann0036/iam-dataset`, ~19k entries) to produce the canonical IAM action string(s). One SDK operation can require multiple IAM actions (e.g. `S3.CopyObject` requires `s3:GetObject` on the source and `s3:PutObject` on the destination); all must be permitted for the request to proceed.

The "carve as you go" model — recording first, then switching to enforcement — inverts the traditional least-privilege workflow. Instead of authoring a policy before running the agent (which requires predicting what the agent will do), you run the agent in recording mode, observe its real behavior, and derive the policy from that. The resulting allowlist reflects actual usage, not a guess.

Because the allowlist is standard IAM policy JSON, it is also directly usable as a session policy scoped at `AssumeRole` time, as an inline role policy, or as input to other IAM tooling — the proxy enforcement layer and AWS IAM are speaking the same language.

### Why this is different from AWS session policies

Session policies scope `AssumeRole` calls at the time of credential issuance. They require the intermediate role to permit self-assumption in its trust policy — which IAC roles do not. They also require the policy author to know in advance what the agent will need.

The proxy allowlist operates per-request at signing time, after credential issuance. It does not require trust policy modifications. It can be updated dynamically without re-issuing credentials. And because it's derived from recorded behavior, it reflects what the agent actually does rather than what someone anticipated.

## Behavioral least privilege

The goal of least privilege in an agent context is: *the agent can do what it needs to do, and nothing else*. Static IAM policies fail at this because they're authored before the agent runs and over-approximate to avoid breaking things.

The recording/enforcement model achieves behavioral least privilege:

1. Run the agent in recording mode against a representative workload.
2. Review the log. Remove any calls that look surprising or unnecessary. Add any calls you know the agent will need that didn't appear in this run.
3. Switch to enforcement mode with the reviewed allowlist.
4. Any new API call the agent attempts — whether from legitimate new behavior, a changed prompt, or a prompt injection — hits the enforcement boundary. It either blocks with an error or requires human approval.

The allowlist captures what the agent *did*, not what it theoretically *could* do. The enforcement boundary catches deviation from that baseline.

## Error forgery for blocked requests

When the proxy blocks a request in enforcement mode, it returns a well-formed AWS error response — the same JSON structure that AWS returns for `AccessDenied` errors — rather than a proxy-level error or a connection reset. This matters because:

- AWS SDKs and the CLI have specific handling for `AccessDenied`. Proxy-level errors (HTTP 502, connection refused) often trigger retry loops or unhelpful error messages.
- Forging a correct `AccessDenied` response means the agent's error handling path executes as designed, not as a surprise.
- The agent cannot distinguish a proxy-enforced denial from a real IAM denial. This is intentional — the proxy's enforcement is meant to be indistinguishable from IAM policy from the agent's perspective.

Example forged response body:

```json
{
  "Error": {
    "Code": "AccessDenied",
    "Message": "User is not authorized to perform: s3:GetObject on resource: arn:aws:s3:::bucket/key"
  }
}
```

The HTTP status code is 403, matching AWS's real behavior.

## KMS-based client identity (aspirational)

The current model uses a proxy-generated random secret as the SigV4 signing key. The root of trust is "the proxy generated this and holds the secret." That's fine locally but has no external verifiability — anything with access to the socket can get a keypair.

A stronger model derives the proxy secret from KMS using the active upstream session:

1. The upstream session has access to a KMS key.
2. When generating a new proxy keypair, the proxy calls `KMS GenerateDataKey` using the upstream session. The returned plaintext key becomes the `secret_access_key` material.
3. The proxy holds the plaintext key in memory for local SigV4 validation. The KMS-encrypted ciphertext is stored alongside it.
4. To verify that a keypair was legitimately issued (e.g. after a restart), the proxy can re-decrypt the ciphertext via KMS — which requires the upstream session to be valid.

This ties proxy keypair issuance to an active, valid upstream session. Rotation is still just a KMS `GenerateDataKey` call. Token expiry can be tied to session validity rather than a separate timer.

Not implemented. This is the direction to go if the "proxy generates random secret" trust model is insufficient for the deployment context.

## Non-goals

- This is not a general-purpose AWS proxy. It does not handle SigV4a (multi-region access points), streaming payload signing (large S3 uploads), or response inspection.
- It does not replace CloudTrail. The proxy log and CloudTrail are complementary: CloudTrail records what AWS authorized, the proxy log records what the agent attempted (including blocked calls that never reach CloudTrail).
- It does not address prompt injection prevention. Prompt injection may cause the agent to attempt calls it shouldn't. The enforcement boundary catches those attempts; it does not prevent the agent from being injected in the first place.
